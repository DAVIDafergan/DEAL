## 2. מדדי ביצועים — לפני/אחרי (מדוד, לא מוערך)

| מדד | לפני | אחרי |
|---|---|---|
| שאילתת נכס בודד ב-DB (בקנה מידה של ~9K יחידות, נמדד עם `SET profiling=1`) | 35ms | 1-2ms |
| Bundle JS ראשי (gzip) | 177KB | 132KB |
| בקשות HTTP לעמוד השוואה עם 4 מועדפים | 4 (מקביליות) | 1 |
| `networkidle` לעמוד השוואה (כולל תגלית סקריפט Travelpayouts שבור מהעולם הישן) | עד ~9.8s | ~3-4s |
| `line-height` בסיסי באתר | לא מוגדר (~1.15 ברירת מחדל דפדפן) | 1.6 |

כל המדדים נמדדו ישירות (EXPLAIN, MySQL profiling, Navigation Timing API, גודל build בפועל) —
לא הרצת Lighthouse מלאה בשלב הזה, מהסיבה שתועדה כבר ב-STEP9-REPORT.md: הסביבה הזו (sandbox
עם תחרות CPU בין תהליכים) נותנת מספרי Lighthouse לא אמינים (אותה קונפיגורציה בדיוק החזירה
67 מול 12 בשתי הרצות עוקבות). מדידה ישירה של מנגנון-השורש (שאילתת DB, גודל bundle, זמן
בקשת רשת אמיתית) עדיפה ואמינה יותר בסביבה הזו.

## 3. שיטת האימות

כל שלב: build אמיתי, שרת `vite preview`/`node --watch`, Playwright headless עם צילומי מסך
בעברית/אנגלית ובשתי רזולוציות, ובמקרים הרלוונטיים — קריאות API ישירות מול MySQL מקומי כדי
לוודא שהדאטה שנכתבה נכונה בפועל (לא רק שה-UI לא קרס). באגים אמיתיים שנמצאו ותוקנו תוך כדי
העבודה (לא תיאורטיים):

- **N+1 אמיתי בשליפת נכס בודד** — `EXPLAIN` הראה סריקת טבלה מלאה; אומת מול DB מקומי שהוזרק
  ל-9,000 יחידות.
- **`createProperty` הפיל בשקט את `nearby_attractions`/`view_type`** בשמירה הראשונה של נכס
  חדש (רשימת עמודות INSERT קשיחה שלא עודכנה).
- **שכפול נכס יצר יחידה מיותרת** — `createProperty` יוצר יחידת ברירת-מחדל משלו; תוקן.
- **סקריפט Travelpayouts שבור** (שריד מעולם הטיסות) נמצא תוך כדי מדידת ביצועים — טען chunk
  נוסף ודיווח לשירות מעקב חיצוני בכל טעינת עמוד; הוסר.
- **"0 בעל נכס"/כפילויות שם**, מירוצי StrictMode וכו' — כל אלה מ-9.x, לא רלוונטיים לשלב הזה.
